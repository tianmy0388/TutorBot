# TutorBot 需求分析与赛题追踪

赛题依据：[中国软件杯 A3《基于大模型的个性化资源生成与学习多智能体系统开发》](https://www.cnsoftbei.com/content-3-1286-1.html)。需求映射以赛题官网当前公开内容为准。

## 1. 项目背景

高校学生常面对资源数量多但质量不一、难以判断先修顺序、统一内容与个人基础不匹配等问题。TutorBot 以一门完整课程为边界，通过对话构建学生画像，由多智能体生成和审核资源，再根据练习与行为数据更新画像、路径和推荐。

## 2. 目标用户与痛点

| 用户 | 主要任务 | 关键痛点 |
|---|---|---|
| 本科生 | 补齐知识、完成课程与项目 | 不知道先学什么，资料难度不匹配，缺少即时反馈 |
| 教师 | 观察薄弱点并进行干预 | 难以持续追踪每位学生，干预缺少证据 |
| 课程建设者 | 建设可信的多模态资源 | 生成内容来源不透明，维护成本高 |

## 3. 功能需求

| 编号 | 需求 | 验收标准 | 对应实现 |
|---|---|---|---|
| FR-01 | 对话式画像 | 自然语言抽取不少于 6 维特征；对话与练习后版本递增 | `profile` capability、ProfileBuilder |
| FR-02 | 多智能体资源生成 | 至少 5 类资源；不同 Agent 有结构化输入输出 | `resource_generation` capability |
| FR-03 | 个性化路径 | 使用知识掌握、先修关系和偏好生成可解释顺序 | KnowledgeGraphPlanner |
| FR-04 | 精准推送 | 同主题对不同画像产生不同类型、难度或顺序 | ResourcePlan、画像快照 |
| FR-05 | 智能辅导 | 即时文字答疑，支持图示、代码、练习、参考或视频补充 | `tutoring` capability |
| FR-06 | 学习评估 | 跟踪练习、学习行为与反馈，输出多维分数和下一步策略 | `assessment` capability |
| FR-07 | 来源说明 | 学生可按需查看引用与仍需确认的说明 | Resource source details |

## 4. 非功能需求

| 编号 | 需求 | 验收标准 |
|---|---|---|
| NFR-01 | 可用性 | 支持流式输出、Markdown、多模态卡片和明确错误状态 |
| NFR-02 | 可靠性 | 长任务持久化；刷新可恢复；部分失败可定向重试 |
| NFR-03 | 性能 | 首次反馈不超过 2 秒；长任务持续显示阶段；记录端到端耗时 |
| NFR-04 | 安全 | 密钥不回传；文件和代码执行受限；敏感内容可阻断 |
| NFR-05 | 可追溯 | 每个任务具有任务 ID、阶段事件、工件、来源和终态合同 |
| NFR-06 | 可部署 | Windows 一键启动；常规 Python/Node 环境可手动启动 |

## 5. 个性化验证案例

对同一主题“Transformer 注意力机制”构造两类学生：视觉型初学者优先获得图示、动画、基础练习和短资源块；代码型进阶者优先获得 NumPy/PyTorch 实验、论文阅读和挑战题。保持主题、模型和知识库不变，对资源类型、难度、顺序和推荐理由进行对比，证明差异来自画像而非随机输出。

## 6. 需求边界

本版本聚焦单学生学习闭环，不包含完整教务系统、多租户收费、班级权限和正式成绩认证。

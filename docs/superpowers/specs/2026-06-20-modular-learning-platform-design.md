# Tutor 模块化个性化学习平台改造设计

## 1. 目标与范围

本轮将现有 Tutor 改造成可稳定完成比赛或答辩演示的单用户系统。系统必须完整展示“对话构建画像 → 选择学习任务 → 个性化资源规划与确认 → 多智能体生成 → 质量与事实审核 → 学习路径 → 效果评估与调整”的闭环。

本轮不实现账号登录、多人协作、租户权限、云端密钥托管、分布式任务队列或生产级高可用。运行环境为本地 Next.js + FastAPI + SQLite，模型 API Key 允许由服务端写入项目根目录 `.env`。

## 2. 已确认的产品决策

- 采用模块化演示闭环，不做局部堆叠，也不全量重写。
- 默认入口为学习工作台，另设知识库、资源中心和设置三个一级页面。
- 普通问答走快速答疑路径，默认不生成视频、PPT 或完整资源包。
- 资源生成前先由系统推荐资源类型，用户可勾选增删并确认。
- 首版知识库上传支持 PDF、DOCX、PPTX、Markdown 和 TXT。
- LLM、Embedding 与 Web Search 均可在网页配置和测试。

## 3. 现状诊断

### 3.1 “任务完成但无输出”的根因

异步 job 提交路径 `ChatComposer → useJobQueue.submit/subscribe` 没有创建对应的前端 active turn。随后 `dispatchStreamEvent` 调用的 `applyStreamEvent` 会在 turn 为 idle 时忽略流事件。收到 `done` 后，`completeActiveTurn` 又先清空全局 `currentCapability`，补发完成消息时只能得到 `unknown`；其“是否已有消息”判断以 `started_at = 0` 比较时间戳，会把历史助手消息误判成本轮消息。因此后端 job 可以在 SQLite 中以 `completed` 保存完整 result，前端却没有任何新增输出。

根因不是 LLM 没有返回，而是异步任务模型与前端单一 `activeTurn` 状态模型不一致。

### 3.2 其他结构性问题

- 一个全局 active turn 无法正确表达多个后台 job、断线恢复或历史重放。
- 当前能力选择器把用户意图过早固定为 capability，缺少“先规划、再确认”的资源生成步骤。
- 资源类型虽然有独立 viewer 和部分生成 Agent，但页面和任务协议仍以整个资源包为中心，单项失败、单项重试和成本提示不清晰。
- 知识库仅为目录中的预置课程，没有统一的列表、上传、摄取状态和索引管理 API。
- 配置服务已有 Settings、EnvStore 和 ModelCatalog 基础，但没有安全的网页 API、密钥掩码、热更新和连接测试。
- 后端测试在当前默认 Python 环境中因依赖缺失及 `pydantic-settings` 版本不兼容无法收集，开发环境不可复现。

## 4. 推荐架构

### 4.1 系统边界

系统保留现有五层结构：Next.js 展示层、FastAPI 应用与编排层、Capability 层、Agent 层、工具与服务层。改造重点是增加三个明确的应用服务边界：

1. `IntentRouter`：区分画像对话、普通答疑、资源生成、路径规划和效果评估。
2. `ResourcePlanService`：根据意图、画像、知识库命中和成本规则生成可编辑的资源清单。
3. `JobService`：以 job 为唯一事实源保存状态、阶段、事件、结果、警告和错误。

### 4.2 两条执行路径

快速路径用于普通问答和画像对话：意图路由后调用知识库检索与答疑 Agent，立即流式输出文字、引用和画像增量。该路径不调用 Manim、PPT 或资源包生成。

生产路径用于个性化资源生成：需求理解后先返回推荐资源计划，进入 `awaiting_confirmation`；用户确认后，各资源类型 Agent 并行生成，随后执行质量审核、事实核查、资源包持久化和学习路径整合。

视频只在用户明确勾选或明确提出视频/动画需求时进入计划，不能仅因“原理”“解释”等宽泛关键词自动触发。

### 4.3 资源模块边界

每个资源类型实现统一生成接口，输入为 topic、profile snapshot、knowledge context 与 type-specific options，输出为一个 artifact result：

- `document`：课程讲解文档；
- `mindmap`：知识点思维导图；
- `exercise`：分层题库；
- `reading`：带来源的拓展阅读；
- `code`：代码实操案例；
- `ppt`：可下载课件；
- `video`：Manim 视频或动画。

每个类型独立记录状态和错误。整个 job 在全部成功时为 `succeeded`，部分成功时为 `partial`，全部失败时为 `failed`。用户可以只重试失败类型。

## 5. Job 状态与事件契约

### 5.1 状态机

资源任务状态依次为：

`planning → awaiting_confirmation → queued → running → succeeded | partial | failed | cancelled`

普通答疑不需要确认，可从 `planning` 直接进入 `running`。

### 5.2 结果契约

每条事件和最终结果必须绑定 `job_id` 与 `capability`。最终响应至少包含：

- `job_id`、`capability`、`status`；
- `assistant_message`，保证终态始终有可见聊天内容；
- `progress`，包含当前阶段、百分比和活跃 Agent；
- `artifacts[]`，包含成功生成的资源；
- `warnings[]`，包含未验证声明、降级或部分失败；
- `error`，包含稳定错误码、用户消息、诊断信息和 `retryable`；
- `event_cursor`，用于断线续传。

前端以 `jobsById` 保存任务，不再用一个全局 active turn 承载所有任务。事件按 `event_id` 去重，按序号合并。页面刷新后先通过 REST 获取 job 快照，再从最后事件序号恢复 WebSocket 订阅。

### 5.3 终态展示

- `succeeded`：聊天展示总结，资源卡可直接打开。
- `partial`：明确列出成功项和失败项，并提供“仅重试失败项”。
- `failed`：展示可理解原因、诊断 ID 和重试入口；配置错误直接链接到设置页。
- `cancelled`：保留已生成的安全产物，并说明任务已停止。

## 6. 页面信息架构

### 6.1 学习工作台

学习工作台由对话与任务区、当前学习路径、任务进度和画像摘要组成。用户输入后，系统先显示识别出的任务类型。普通答疑立即流式回答；资源生成显示推荐清单、预计耗时和昂贵类型提示，确认后才执行。

### 6.2 知识库

知识库页展示所有课程库及其文档数、分块数、Embedding 模型、索引状态、更新时间和错误。用户可创建知识库、上传文件、选择当前知识库、重试失败摄取和删除文档。

文件摄取状态为 `uploaded → extracting → chunking → embedding → ready | failed`。原文件、抽取文本、元数据与索引均按 knowledge base ID 隔离。无法解析、空文档、超限和不支持格式必须返回明确错误。

### 6.3 资源中心

资源中心按课程、资源类型、状态和时间筛选资源包。用户可以预览、下载、查看参与 Agent、引用与审核报告，也可以重新生成单项资源。资源详情继续复用现有文档、思维导图、题库、代码、PPT 和视频 viewer。

### 6.4 设置

设置页分为外观、LLM、Embedding、Web Search 四组。模型配置包含 provider、model、base URL、API Key 和必要的推理参数。Web Search 展示当前支持的 provider 与对应参数。

读取配置时 API Key 只返回“是否已配置”和掩码；更新时空值表示保留旧密钥，显式清除使用独立字段。保存后由服务端原子更新根目录 `.env`，清理 Settings 缓存并重建相关 provider。每组提供测试按钮，返回成功状态、耗时、实际 provider/model 和脱敏错误。

## 7. 知识库与 RAG

预置的 `ai_introduction` 继续作为完整高校课程样例。上传文件通过统一 loader 转为带来源锚点的规范文档，再按设置中的 chunk 参数分块并调用当前 Embedding provider 建索引。

答疑和资源生成都必须显式接收选中的 knowledge base ID。检索结果携带文档名、页码或段落锚点、相似度与文本片段。没有可用索引时允许降级为通用模型回答，但界面必须标记“未使用课程知识库”。

## 8. 防幻觉与内容安全

- 输入端执行敏感内容分类和文件类型校验。
- 学术事实优先由知识库片段支撑；开启 Web Search 时可补充外部来源。
- 生成后执行引用覆盖检查、声明抽取、事实核查和内容安全审核。
- 无法验证的内容不删除事实状态，而是标为 `unverified` 并降低置信度。
- 每个资源展示引用、置信度、审核结论及参与 Agent，使“多智能体”和“防幻觉”在答辩界面中可见。

## 9. 响应性能与进度

- 提交后 1 秒内出现 job 卡或首个流式反馈。
- 每个阶段发送开始、进度、完成或失败事件，并显示活跃 Agent。
- 文档、思维导图、习题、阅读和代码可并行执行。
- 视频与 PPT 作为独立长任务展示进度，不阻塞其他成功资源的查看。
- WebSocket 断线不改变后台 job 状态，恢复后继续订阅。

## 10. 测试与验收

### 10.1 自动化测试

- Intent router：普通问答不触发视频或资源生产。
- Resource planner：画像偏好影响推荐，视频仅在明确选择时执行。
- Job state machine：成功、部分成功、失败、取消、重放和去重。
- Result contract：任何终态都包含可见的 `assistant_message`。
- Config API：密钥掩码、保留、清除、原子更新、缓存刷新和连接测试。
- Knowledge ingestion：五种格式、空文件、损坏文件、重复文件和失败重试。
- Safety：引用覆盖、未验证标记与敏感内容拒绝。

### 10.2 端到端场景

1. 通过自然对话构建并更新不少于六维学习画像。
2. 普通答疑使用知识库引用并确认没有视频任务。
3. 资源生成先推荐类型，用户增删后生成至少五类资源。
4. 某一资源失败时 job 为 partial，成功资源可用且失败项可单独重试。
5. 上传课程文件后可在知识库列表看到索引进度，并可被答疑引用。
6. 错误 API Key 测试返回明确错误，修改配置后无需重启即可恢复。
7. 刷新页面或模拟断线后任务进度与结果可以恢复。
8. 完成练习和评估后，画像、学习路径与推荐发生可解释的更新。

## 11. 开源与文档要求

README 和显著位置继续列出 DeepTutor、ManimCat、Manim Community、LlamaIndex、Next.js、Tailwind CSS 等项目的名称、来源和许可证。新增文件解析或向量存储依赖时同时补充来源和协议。

## 12. 实施分解

实施按可独立验收的顺序拆分：

1. 修复 job 状态与无输出回归，并建立前端测试基础。
2. 建立意图路由与资源计划确认协议。
3. 将资源类型改造成独立 artifact 任务并支持 partial/retry。
4. 增加配置 API、设置页与连通性测试。
5. 增加知识库模型、文件摄取 API 和知识库页。
6. 重构一级导航、学习工作台和资源中心。
7. 串联画像、路径、评估、安全证据和完整答辩场景。
8. 修复依赖锁定、自动化测试、文档与演示数据。
